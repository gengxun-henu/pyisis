## 我怎么知道“哪些是优先的”

我现在查到的优先级来源主要有 3 层：

1. pybind_progress_log.md  
   这是**人工维护的进度日志**，记录每一批做了什么、测了什么、还有哪些 blocker。

2. methods_inventory_summary.csv  
   这是**自动生成的总表**，里面有：
   - `Suggested Priority`
   - `Priority Rank`
   - `Open Items`
   - `Completion Percent`
   - `Priority Reason`

3. `isis_pybind_standalone/class_bind_methods_details/*_methods.csv`  
   这是**单类明细表**，会细到每个方法是 `Y / N / Partial`。

### 这个优先级不是拍脑袋，是脚本算出来的

在 generate_methods_csv.py 里，优先级规则写得很明确：

- 如果类**已经导出到 Python**：
  - `open_items <= 10` → `High`
  - `open_items <= 30` → `Medium`
  - 否则 `Low`

- 如果类**还没导出**：
  - `total_items <= 12` → `High`
  - `total_items <= 30` → `Medium`
  - 否则 `Low`

并且排序规则是：

1. `Priority Rank`
2. `Open Items` 越少越前
3. `Completion Percent` 越高越前
4. 再按模块 / 类名

所以如果你只说一句：

> 按高优先级继续做

我默认会按这个工作流选：

- 先挑 `High`
- 再挑 **Open Items 少**
- 优先挑 **已经有绑定、只差几个方法** 的类
- 或者 **总 API 很小、容易快速补完** 的未绑定类
- 再额外看是否有**现成测试模式**可以复用

### 必更：进度 / 覆盖率追踪文件

这是当前工作流里明确要求要维护的。

#### 1. pybind_progress_log.md
这是**必须优先更新**的进度记录。

通常会补这些信息：

- 处理了哪个类 / 哪个模块
- 改了哪些 binding / test 文件
- 做了哪些验证

这个要求在技能说明和 progress reference 里都写得比较死。

#### 2. 对应类的 `*_methods.csv`
如果这次任务涉及覆盖率同步，我会更新对应类的明细 CSV，反映：

- 哪些方法从 `N` 变成 `Y`
- 哪些还是 `Partial`
- 备注里写清边界

#### 3. methods_inventory_summary.csv
这个一般不是手填，而是由  
generate_methods_csv.py  
**重新汇总生成**。

也就是说，它更像是**派生文档**，不是主编辑对象。

## 一般也会改，但它们不算“文档”

### 4. __init__.py
如果新增了需要包级导出的类或函数，这里通常会同步加 export。

这不是“说明文档”，但它是 Python API 对外可见面的重要一层。

### 5. 测试文件
通常会补：

- `isis_pybind_standalone/tests/unitTest/*.py`
- smoke_import.py

这也不是文档，但它们是“可验证记录”。

## 哪些文档不会默认每次都动

这些文件**不一定每次都会更新**：

- README.md
- testing.md
- packaging.md

我只有在下面情况才会主动建议更新它们：

- 用户可见的 API 使用方式发生明显变化
- 构建 / 测试流程改变了
- 新增了值得在 README 公开说明的大模块
- 某个常见坑已经反复踩到，适合沉淀到文档里

换句话说：

- **进度文档**：默认更新
- **覆盖率文档**：默认同步
- **README 类文档**：按需更新，不是每次都机械改

## 我是靠什么机制做到这些同步的

不是“我天然会记住一切”，而是靠这套**可追踪工作流**：

### 机制 1：固定入口文件

每次 pybind 工作默认先看：

- todo_pybind11.csv
- methods_inventory_summary.csv
- pybind_progress_log.md
- 目标类 `*_methods.csv`

这是仓库技能说明里明确规定的起手动作。

### 机制 2：自动优先级脚本

generate_methods_csv.py 把：

- 当前 todo 状态
- 头文件 API
- binding 源码扫描结果

合成成单类 CSV 和 summary CSV。  
所以优先级不是纯人工维护，而是**自动推导**出来的。

### 机制 3：源码交叉校验

因为自动表可能滞后，我还会检查：

- `isis_pybind_standalone/src/**/*.cpp`
- `python/isis_pybind/__init__.py`
- `tests/smoke_import.py`
- 附近已有 unit test

也就是说，**文档是导航，源码是最终真相**。

### 机制 4：任务完成后的状态回写

完成一个 binding 任务后，按工作流要把结果回写到：

- pybind_progress_log.md
- 需要时回写 todo_pybind11.csv
- 需要时回写 `*_methods.csv`
- 重新生成 / 同步 methods_inventory_summary.csv

这是一个“做完 → 验证 → 回填状态”的闭环。
