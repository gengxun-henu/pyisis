## 1. 产品概述

本产品旨在为行星遥感与摄影测量场景提供一个网页端在线处理平台。用户无需手工组织复杂的 ISIS 命令链，只需上传原始影像、选择少量参数，即可在网页端完成从原始数据导入、SPICE 初始化、控制网构建、光束法平差到正射成果输出的完整处理流程。

首期重点聚焦 LRO NAC 原始 PDS `IMG` 数据，后续逐步扩展到 CTX、HiRISE、Apollo、Kaguya 等 mission 数据。

## 2. 产品目标

### 2.1 业务目标

1. 降低 ISIS 使用门槛，让非命令行专家也能完成标准摄影测量处理。
2. 将当前仓库已有的 `isis_pybind` 与 `examples/controlnet_construct/` 工作流转化为在线服务能力。
3. 为单景与多景处理提供统一的任务视图、日志视图和成果视图。
4. 支持最终成果下载与百度网盘归档。

### 2.2 MVP 目标

首期 MVP 只要求完成以下核心链路：

1. 支持上传单景或多景 LRO NAC 原始 `IMG`。
2. 支持创建异步处理任务。
3. 支持单景流程：ingest → `spiceinit` → 正射输出。
4. 支持多景流程的任务占位：重叠分析、控制网生成、bundle、正射结果输出。
5. 支持网页端查询任务状态、步骤日志、结果清单。
6. 支持成果归档到百度网盘的异步占位流程。

## 4. 核心使用场景

### 场景 A：单景快速正射

用户上传 1 幅 LRO NAC 原始 `IMG`，平台自动完成：

1. 导入到 ISIS cube
2. `spiceinit`
3. 校正与地图投影
4. 生成正射成果
5. 页面提供日志、下载与归档状态

### 场景 B：多景控制网构建

用户上传多幅相邻影像，平台自动完成：

1. 影像预处理
2. 重叠关系识别
3. DOM matching 控制网构建
4. pairwise ControlNet 合并
5. 输出控制网与统计摘要

### 场景 C：多景联合平差

用户在控制网基础上进一步选择“执行光束法平差”，平台自动完成：

1. 调用 `jigsaw`
2. 生成平差报告
3. 基于平差结果重新输出几何纠正成果
4. 展示质量评估摘要

### 5.1 In Scope（首期范围）

## 6. 功能需求

### 6.1 文件上传

#### FR-001 上传原始数据

系统应支持上传单个或多个原始影像文件。

#### FR-002 上传任务分组

系统应允许将一组上传文件归入同一个 processing job。

#### FR-003 基础校验

系统应校验：

1. 文件是否存在
2. 文件扩展名是否允许
3. 同一任务内文件名是否重复
4. 文件大小是否超出限制

### 6.2 任务创建

#### FR-010 创建任务

系统应允许用户创建两类任务：

1. `single_scene`
2. `multi_scene_bundle`

#### FR-011 参数最小化

创建任务时，页面默认仅暴露少量参数：

1. 目标体
2. DEM / shapemodel 选择
3. 控制网生成方式
4. 是否执行 bundle adjustment
5. 输出投影模板

#### FR-012 任务入队

任务创建成功后，系统应将其写入数据库并送入异步队列。

### 6.3 单景处理

#### FR-020 ingest

系统应支持将 LRO NAC 原始 `IMG` 转换为 ISIS cube。

#### FR-021 SPICE 初始化

系统应支持对导入后的 cube 自动执行 `spiceinit`。

#### FR-022 正射输出

系统应支持执行地图投影或正射输出，并产出最终成果文件。

### 6.4 多景处理

#### FR-030 重叠分析

系统应支持在多景任务中识别候选重叠影像对。

#### FR-031 控制网构建

系统应支持复用 `examples/controlnet_construct/` 进行 pairwise ControlNet 构建。

#### FR-032 控制网合并

系统应支持生成 merge 所需输入并产出 merged ControlNet。

#### FR-033 光束法平差

系统应支持在控制网基础上执行 `jigsaw` 占位工作流。

#### FR-034 平差后成果输出

系统应支持平差后成果重生产的任务占位。

### 6.5 任务监控

#### FR-040 查询任务状态

系统应返回任务级状态：

- `created`
- `queued`
- `running`
- `succeeded`
- `failed`
- `cancelled`
- `archiving`
- `archived`

#### FR-041 查询步骤状态

系统应返回步骤级状态与顺序信息。

#### FR-042 查看日志

系统应支持读取步骤日志与 JSON 摘要。

### 6.6 成果与归档

#### FR-050 成果列表

系统应展示每个任务的成果清单。

#### FR-051 下载能力

系统应为成果文件生成下载入口。

#### FR-052 百度网盘归档

系统应支持将最终成果异步归档到百度网盘，并记录归档状态。

## 7. 非功能需求

### NFR-001 可追踪性

每个任务必须保留：

1. 输入文件列表
2. 参数 JSON
3. 步骤状态
4. 日志路径
5. 输出文件路径

### NFR-002 失败可定位

任务失败时必须至少能定位到具体步骤和错误摘要。

### NFR-003 异步处理

所有处理任务必须通过异步 worker 执行，避免 API 阻塞。

### NFR-004 可扩展性

后端结构必须支持后续扩展更多 mission、更多 worker 类型和更复杂 DAG。

### NFR-005 环境隔离

算法执行层应与 API 层解耦，便于后续使用容器化 worker 节点。

## 8. 用户流程

### 8.1 单景任务流程

1. 用户上传原始影像
2. 用户选择最少参数并创建任务
3. 系统入队
4. worker 执行 ingest / `spiceinit` / 正射
5. 用户查看步骤进度和日志
6. 用户下载成果或查看归档状态

### 8.2 多景任务流程

1. 用户批量上传影像
2. 用户创建多景任务
3. 系统执行 overlap / controlnet / merge / bundle / orthorectify
4. 用户查看各步骤日志、摘要和成果清单
5. 用户下载成果或等待网盘归档

## 10. 后端原型映射

本 PRD 对应的后端原型骨架建议放在：

- `python/online_platform/`

建议包含：

1. `app.py`：FastAPI 应用入口
2. `config.py`：环境配置
3. `api/routes/`：健康检查、上传、任务接口
4. `services/`：作业服务、存储服务、归档服务
5. `tasks/`：Celery app 与工作流构建
6. `domain/`：任务、步骤、成果等内部数据模型

## 11. MVP 验收标准

### 单景 MVP 验收

1. 可以创建单景任务
2. API 能返回任务状态
3. worker 能执行占位单景流程
4. 可以看到步骤日志和结果占位

### 多景原型验收

1. 可以创建多景任务
2. 可以生成多景步骤 DAG 占位
3. 可以看到 controlnet / bundle / archive 等步骤的状态占位

## 13. 一句话总结

本 PRD 的核心是：

> 在不重写 ISIS 算法内核的前提下，把当前仓库已有的 `isis_pybind` 与 DOM Matching ControlNet 工作流封装成一个可提交任务、可异步执行、可追踪结果、可归档成果的网页端在线处理平台。
